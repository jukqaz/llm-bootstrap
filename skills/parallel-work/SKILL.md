# 병렬 작업 분리

## Overview

동시에 여러 작업을 진행할 때 작업 스트림을 분리하고 문맥 혼선을 줄여라. 기본 전략은 브랜치 + 작업 디렉터리 분리이며, Git 사용 시 `git worktree`, Jujutsu 사용 시 `jj workspace`를 우선 고려하라.

## Workflow

### 1) 작업 스트림 정의
- 병렬로 처리할 작업 목록, 우선순위, 마감, 의존성을 확인하라.
- 작업 간 파일/모듈 충돌 가능성이 있으면 병렬 처리 여부를 재검토하라.
- 정보가 부족하면 먼저 질문하라.

### 2) 분리 전략 선택
- Git을 쓰면 `git worktree`를 기본으로 선택하라.
- Jujutsu를 쓰면 `jj workspace`를 기본으로 선택하라.
- 의존성/런타임이 크게 다른 작업이면 별도 클론을 검토하라.
- 사용 도구가 불명확하면 먼저 확인하라.

### 3) 스트림별 계획 정리
- 각 스트림을 `pre-work-plan` 흐름으로 계획하라.
- 복잡 작업이면 Linear 이슈 생성 여부를 확인하라.
- PRD/TRD가 없으면 `project-planning-docs` 사용을 제안하라.

### 4) 분리 실행
- `references/git-worktree.md` 또는 `references/jj-workspace.md`를 필요할 때 읽어라.
- 새 브랜치/워크스페이스 생성은 사용자 승인 후 진행하라.
- 파괴적 정리는 항상 확인을 받은 뒤 수행하라.

### 5) 병렬 작업 규칙
- 스트림마다 작업 디렉터리/브랜치를 고정하라.
- 한 스트림에서 다른 스트림 파일을 수정하지 말아라.
- 커밋은 스트림 단위로 분리하고 메시지는 명확히 작성하라.

### 6) 마무리/통합
- 작업 완료 후 브랜치를 정리하고 필요하면 워크트리를 제거하라.
- 문서 변경이 있으면 `docs-audit` 또는 `doc-linker` 사용을 검토하라.
- 배포/릴리즈 문서가 필요하면 `release-docs` 사용을 제안하라.

## MCP/도구 연계

- Linear MCP가 설정되어 있으면 스트림별 이슈로 추적하라.
- GitHub MCP가 설정되어 있으면 스트림별 PR 생성 시 활용하라.
- 설정 여부가 불명확하면 사용자에게 먼저 확인하라.

## Output checklist

- 병렬 작업 목록과 분리 전략이 합의되었다.
- 스트림별 계획(또는 Linear 이슈)이 정리되었다.
- 작업 디렉터리/브랜치가 충돌 없이 분리되었다.
- 마무리 시 문서/릴리즈 연계가 검토되었다.
