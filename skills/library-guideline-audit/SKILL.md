# Library Guideline Audit

라이브러리나 프레임워크 사용 방식이 현재 공식 권장사항과 맞는지 확인할 때 사용한다.

## Overview

범위: 공식 최신 문서 기준 감사, mismatch 분류, 수정안 제시, 교체 필요성 판단, 변경 이유 설명
비범위: 대규모 코드베이스 단순화(code-polish-simplify), 변경 후 후속 점검(change-completeness)

## Workflow

### 1) 기준 문서 확정
- 공식 사이트, 공식 문서, 공식 migration guide를 source of truth로 잡아라.
- 버전 차이가 있으면 현재 사용 버전과 목표 버전을 먼저 구분하라.
- 비공식 블로그나 예제는 보조 참고로만 써라.

### 2) 현재 구현 감사
- 현재 코드가 어떤 API, 패턴, 옵션을 쓰는지 빠르게 목록화하라.
- 권장사항과 다른 지점을 찾아 `유지 가능`, `권장 수정`, `교체 검토`, `즉시 수정 필요`로 분류하라.
- 단순 최신화와 가이드라인 불일치를 혼동하지 말아라.

### 3) 수정안과 교체 판단
- 가능한 한 현재 라이브러리를 유지하면서 권장 패턴으로 맞추는 안을 먼저 제시하라.
- 교체를 제안할 때는 성능, 유지보수성, 지원 중단, 보안, 공식 권장 경로 같은 이유를 명시하라.
- “왜 바꾸는지”가 모호하면 교체 제안을 보류하라.

### 4) 영향 범위 정리
- 런타임 영향, 테스트 영향, CI 영향, 문서 영향이 있는지 적어라.
- 후속 수정이 필요한 파일/모듈과 검증 명령을 함께 남겨라.

## Output checklist

- 공식 기준 문서가 명확하다.
- mismatch가 우선순위와 함께 분류되었다.
- 교체 제안이 있다면 이유가 분명하다.
- 후속 테스트/CI/docs 영향이 정리되었다.
